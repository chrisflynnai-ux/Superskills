## HOW TO WORK

### Before Starting Any Task
1. **Read relevant skill XML** — Load L1 always, L2 when executing
2. **Check for SSOT objects** — PROJECT_BRIEF and MESSAGE_SPINE required
3. **Identify awareness level** — Where is the reader entering/exiting?
4. **Lock strategy first** — No generation without locked inputs

### During Execution
1. **Use Python tools for validation** — Deterministic > LLM guessing
2. **Run MMA checks continuously** — Catch drift early
3. **Generate CORRECTION_LOG** for any fixes
4. **Propose PATCH if same failure repeats**

### Delivery Standards
1. **Single CTA** — One action, repeated (not multiplied)
2. **Proof discipline** — All claims grounded or softened
3. **Mobile-first** — Scannable, short paragraphs
4. **No fake urgency** — Only factual or consequence-based

## TOOL GOVERNANCE

### Always Call Tools When:
- A claim needs grounding (proof discipline)
- A schema can be validated (Pydantic/Python)
- A diff/merge is needed (locked objects)
- Context is bloating (summary/GC)

### Avoid Tools When:
- Brainstorming hooks/angles (Draft Track)
- Generating variants intentionally (controlled creativity)

## DEBUGGING

### If Output Quality is Low:
1. Check if SSOT objects are complete
2. Verify skill XML was loaded (L1+L2 minimum)
3. Run MMA validation to identify weak dimension
4. Check for drift from MESSAGE_SPINE
5. Apply circuit breaker if same issue repeats

### If Context is Bloating:
1. Trigger garbage collection
2. Write SUMMARY_HANDOFF
3. Persist RUN_STATE
4. Drop non-essential transcripts
